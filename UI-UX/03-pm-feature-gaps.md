# TheraNotes-AI — Feature Gap Audit (Internal SaaS Lens)

**Scope:** What's structurally missing for 3-5 OTs at a single clinic to rely on this day-to-day. Not visual polish, not clinical workflow specifics.
**Method:** Read `app/`, `components/`, `supabase/migrations/`, `hooks/`, `app/api/`. Confirmed `localhost:3000` boots and the auth/redirect flow exists. Did not click Generate.
**Verdict:** A polished single-screen demo of "paste notes -> generate FCA". Generator and workspace are real; identity, library, configuration, audit, recovery and multi-user are stubbed or absent. To cross from demo to internal app, the surface area outside the generator needs to roughly double.

---

## 1. Account & Identity

**What exists.** Supabase auth via `@supabase/auth-ui-react` `ThemeSupa` widget on `/login`. `UserMenu` shows email + Sign out; avatar is the first two letters of the login email, hardcoded. No `/account`, `/profile`, or `/me` route. No profile state beyond `user.email`.

**Gaps**

| Missing | Why an internal SaaS needs it | Cost | Priority |
|---|---|---|---|
| Display name + credentials separate from login email | `assessor_name` / `assessor_credentials` are retyped per assessment; should live on a profile and prefill | S | P0 |
| Profile page (`/settings/account`) for name, credentials, AHPRA, default email/provider | Anchor for multi-user later; without a `profiles` table the app can never say "Generated by Hayley" | S-M | P0 |
| Avatar (uploaded or generated from name) | Two-letter email slice looks like a debug artefact | S | P1 |
| Email change + password reset UI | Forgot-password flow not wired | S | P1 |
| Session timeout banner | Silent token expiry mid-edit will eat work | S | P1 |
| Sign-up vs invite gating | `/login` allows public sign-up via Supabase Auth UI; clinic tool wants invite-only | S | P0 |
| "Signed in as X" visible outside the dropdown | Reinforces ownership; sets up shareable drafts | XS | P1 |

Combined fix: add a `profiles` table joined on `auth.users.id` (`display_name`, `credentials`, `default_company`, `ahpra`, `avatar_url`, `default_assessment_mode`), prefill `/generate`, render the name in the topbar.

**Gaps**

| Missing | Why | Cost | Priority |

| Template manager: pick which sections, in what order, for which report type | Current single hardcoded JSON | M | P1 |

## 5. Audit Log & Compliance

**What exists.** `generation_logs` (migration `006`) captures system + user prompt, RAG chunks, model, tokens, latency, raw + processed output, status, per-row user RLS. **Zero user-facing UI** for it — it's a developer table. Reports have `coherence_result`, `insufficient_data_flags`, `planner_review` JSON; coherence and flags surface in the workspace TOC/footer, underlying LLM trail does not.

**Gaps**

| Missing | Why | Cost | Priority |
|---|---|---|---|
| Per-report activity timeline ("Generated, Reviewed, Edited 3 sections, Exported DOCX") | Internal tool baseline. Without it, "who changed Part C between Tuesday and Friday" is unanswerable. | M (events table + UI) | P0 |
| "View prompt / view RAG chunks for this section" debug drawer | The data is already logged. Surfacing it would let clinicians self-diagnose ("AI hallucinated because exemplar X was matched"). High leverage from existing infra. | S | P1 |
| Data retention statement + per-report "Permanently delete (PHI)" affordance | Health data. Today: delete a report, the `generation_logs` row cascades but the storage objects from the exemplars don't, and there's no confirmation that prompts contained `clinical_notes` were also wiped. | M (verify cascade + add UI copy) | P0 |
| Region / encryption disclosure that matches reality | Footer claims "Data stays in Australia". Confirm Supabase project is `ap-southeast-2`. If yes, surface it on a privacy page. If no, stop saying it (compliance risk). | XS to verify, M to fix if false | P0 |
| Sign-in audit (last sign-in, IP, device) | Standard. Free from Supabase auth logs but not surfaced. | S | P2 |
| Export audit ("Exported by Hayley on 9 May 14:03 to FCA-John.docx") | Once a report leaves the system, the trail dies. | S | P1 |

Structural insight: the *technical* audit infra (`generation_logs`) is more mature than the *user-facing* audit, which is non-existent. Inverting that unlocks trust cheaply.

## 11. Resilience

**What exists.** Generate is N sequential POSTs to `/api/generate`, one per section, then a coherence call. If section 4 of 8 fails, the loop throws and `setError` shows a red box. The partial report exists in the DB (sections 1-3 saved), but the UI never surfaces "resume". No retry button. `useAutoSave` re-marks dirty on save failure but never tells the user — if Supabase is down for 10 minutes, the editor still says "Saved just now".

**Gaps**

| Missing | Why | Cost | Priority |
|---|---|---|---|
| Resume-from-failure on a half-generated report | The DB has the partial data. The UI does not let you continue. So users restart from scratch and double-spend tokens. | M | P0 |
| Visible save-failure state ("Couldn't save — retry") | Silent autosave failure on health data is unacceptable. | S | P0 |
| Per-section retry on the workspace ("Regenerate Part C") with the original prompt + RAG | The data is in `generation_logs`. Wiring it up should be straightforward. | M | P1 |
| Generate flow guarded by network-failure handler with exponential backoff | Currently a single fetch failure kills the whole report | S-M | P0 |
| Health check / "service is up" indicator | Avoids the "is it the network or me?" question | S | P2 |

## 13. Export Quality

**What exists.** Two export paths, both using the `docx` library: `export-button.tsx` (post-generation, filename `FCA-Report.docx`) and `workspace-layout.tsx` `handleExportDocx` (workspace footer, reads live Plate state, filename `FCA-${participant_name}.docx`). Identical structure: title page, H2 per section with thin grey bottom border, body at size 22 (11pt), `\n\n` paragraph splitting.

What this produces is **a valid DOCX, not a clinical document**:

- No letterhead, logo, clinic name, AHPRA, or signature block.
- No page header/footer (page numbers, "Confidential", "FCA — John Smith").
- No table of contents. Default Calibri/Times margins.
- Markdown content (bold, lists, tables) dumped as plain text — no markdown-to-docx conversion. Bullets become asterisks. `**bold**` stays as literal asterisks.
- No PDF export. "Print" calls `window.print()` and there is no `@media print` block in `globals.css` to hide the workspace chrome.

**Gaps**

| Missing | Why | Cost | Priority |
|---|---|---|---|
| Markdown-to-DOCX (bold, italics, lists, tables) | AI lists print as `* item`; export looks amateur | M | P0 |
| Letterhead, logo, assessor signature block | Stops the report looking like a chatbot transcript | M | P0 |
| Page header + footer (participant, date, X/Y, Confidential) | Clinical baseline | S | P0 |
| TOC from section headings | 8+ page FCAs are unnavigable without it | S | P1 |
| PDF export (server-side, not `window.print`) | Email-shareable, locked formatting | M | P1 |
| Print stylesheet hiding workspace chrome | Current `window.print()` is a comedy result | S | P0 |
| Export presets (NDIS / Internal / Plain) | Different audiences | M | P2 |
| Configurable filename | Today `FCA-${name}.docx` — no date, no version | XS | P1 |

The export is the one artefact that escapes the app. Today it represents the clinic poorly.

## The "Missing 20% That Makes It Feel Real"

In priority order:

1. **Profiles + clinic settings.** `profiles` table; assessor name, credentials, AHPRA, default email/company prefilled into `/generate`. Clinic Settings page with name, ABN, AHPRA, logo upload. Wire into DOCX export. Single change, output stops looking like a chatbot transcript.
2. **Real autosave on /generate + draft model.** Create assessment row on first keystroke (status `draft`), persist notes, restore on refresh. Drafts in the report list. #1 trust killer today.
3. **Search across reports + kill the dead Find button.** Debounced filter against participant_name + ndis_number + sections content.
4. **Hard-delete confirmation that mentions PHI + verified cascades.** Typed-confirmation modal; verify storage cleanup.
5. **Export that handles markdown.** Bold/lists/tables rendered as proper DOCX runs, not literal asterisks. The one fix that makes a 30-second visual difference.
6. **Per-report activity log.** Append-only `report_events` (`generated`, `edited`, `reviewed`, `exported`). Timeline in the workspace right rail.
7. **Resume a half-generated report.** Card shows "Resume" not "Failed"; reuse sections already in DB.
8. **Identity everywhere.** Replace two-letter email avatar with display_name; thread it into topbar, breadcrumbs, DOCX footer.
9. **Cost transparency: `/settings/usage`.** Reports this month, total tokens, est. cost. Trivial on `generation_logs`.

Roughly 1.5-2 weeks of focused work. None of it touches the clinical AI logic.

## What I'd Build First (1 Week of Focused Time)

A single PR-able vertical that makes the app demonstrably "real" to an OT pilot:

**Day 1 — Identity foundation.** Migration `008_profiles.sql` (id FK to auth.users, `display_name`, `credentials`, `ahpra`, `default_company`, `default_assessor_email`, `avatar_url`, RLS own-row). Auto-create trigger on sign-up. `lib/profile.ts` server helper. `UserMenu` uses `display_name` initials. New `/settings/account` page (shadcn Form + Zod).

**Day 2 — Clinic settings + export letterhead.** Migration `009_clinics.sql` (single row keyed to user; org-scope later): `clinic_name`, `abn`, `address`, `logo_url`, `letterhead_color`. `/settings/clinic` page with logo upload to a `clinic-assets` bucket. Both DOCX generators call a new shared `lib/export/build-docx.ts` that adds page header (logo + clinic name), footer (`participant_name · page X/Y · Confidential`), markdown-to-DOCX rendering (`marked` -> AST -> docx, ~80 LOC), and `FCA-${participant_name}-${YYYYMMDD}.docx` filenames. Add `@media print` in `globals.css` to hide workspace chrome.

**Day 3 — /generate autosave + drafts.** First keystroke: insert `assessments` row with `status='draft'`, cache the id. Every change: PATCH debounced 1.5s (reuse `useAutoSave`). On mount: query latest draft < 7 days old, offer "Restore your draft from yesterday". Add `beforeunload` guard. Reports list shows drafts with their own status/filter chip.

**Day 4 — Search + report list polish.** Debounced 200ms search input above the grid (`participant_name`, `ndis_number`, sections text). Status filter chips. Hard-delete modal (type the participant's name). Wire-or-remove the workspace "Find" button. Move the stale-generating reaper out of the client into a Postgres function or server-side check.

**Day 5 — Activity log + resume + identity surfaced**
- Migration `010_report_events.sql`: `report_events (id, report_id, user_id, kind, payload jsonb, created_at)`. Kinds: `generated_section`, `coherence_run`, `review_run`, `edited_section`, `exported_docx`, `created_draft`. Per-`user_id` RLS.
- Insert events from `generate/route.ts`, `review/route.ts`, and workspace `handleExportDocx`.
- Right-rail Activity panel listing events with timestamp + actor display_name.
- Resume-from-failure: if `report.status === 'failed'` and sections exist, show "Resume from Section X" on the report card. Generate flow reads `report.sections` first and skips completed.
- Sprinkle `display_name` into workspace breadcrumb, report card, DOCX footer.

**End of week deliverable.** Pilot OT logs in, sees their name in the topbar, searches "John" in Reports, resumes a draft from yesterday (autosave caught it), sees a timeline of who-did-what, exports a DOCX with real letterhead, real bullets, and real page headers. The product crosses from "demo" to "tool".

Deliberately punted: organisations/multi-user (data model is now clean for it), versioning (one snapshot per export is enough this week), cost dashboard (week-two), mobile (tablets fine, phone not needed yet), command palette (week-three), audit hardening (week-four).

Nothing in the week-one plan touches the clinical AI logic. The generator is good. The shell around it is what's holding it back.
